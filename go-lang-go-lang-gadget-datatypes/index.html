Go Go gadget datatypes!

12 April 2016

When learning a new language, it's often beneficial to start with the variables -- how do we declare them; what is their syntax; how can I relate new data structures to ones I already know; etc. I've been learning Go for the past few months and gone through a lot of ups-and-downs in trying to learn a statically-typed language when all I knew before, JavaScript, was untyped.

What does it mean to be statically typed?

Basically, it means that all variables are assigned types when they are declared, and these types are gospel -- they may never be changed while that variable is still in memory. In JavaScript, variables have no type -- you can assign whatever type of data (integer, string, array, etc) to any variable you like. This paradigm in Go is very difficult to get used to if you are coming from an untyped language.

Variable types throughout the code are checked upon compilation to verify that all variables are adhering to their given types. If you try to do something like

var a int a = "hello world"

your program will blow up when it tries to compile. JavaScript will allow this behavior, and if there ever is an issue with variable types, it will only detect it during run-time.

This can be very frustrating at times, as it limits what you can do with your variables. Functions are declared to accept only specific types, so if you try to send an array of characters to a function that needs a string, ka-blooey. My experiences has involved frequent type-conversions to maneuver my data through the workflow.

However, there are very clear advantages to a typed language. It prevents a lot of small but application-breaking errors since it vets all of your code. It also aids in writing small, re-useable functions, as your functions can be written with 100% confidence in the type of data that will be passed in as an argument. Additionally, there will be no confusion about the type of data returned from the function either.

Comparing data types in Go to JavaScript

Let's look at what each language has to offer.

Boolean and Strings: These behave largely the same in both languages

'Numeric': JavaScript has the type 'number' which can be used to represent any (well, most) positive or negative number. Go has a variety of numeric datatypes; each is tailored to be optimized for the expected numeric information to be stored. Numbers may be signed or unsigned (unsigned numbers cannot be negative) and may include only small numbers (e.g. int8), extend also to large numbers (e.g. int64), or specify the inclusion of decimal values (float32 or float64).

As always, Go requires a higher degree of specificity and rigidity than JavaScript.

Maps (Go) and Objects (JavaScript): These represent the default hash table implementation in their respective languages. They operate very similarly, except that in Go, as you may expect, you are required to specify the data type of both the key and the value when you declare your maps.

Arrays: In JavaScript, array are not restricted by any pre-defined length and may be grown or shrunk as needed. This is not the case in Go -- array lengths must be declared at the outset and cannot be modified.

However, Go does have an additional data type which behaves more closely to arrays in JavaScript known as a Slice. Slices are an abstraction of arrays -- a slice describes a segment of an array by specifying a pointer to the start of the array, the length, and the capacity (maximum length). Even though slices have a specified capacity, Go provides default functions, such as 'append', which allow us to grow our slice past the specified capacity. Append operates much like the 'push' function in JavaScript.

Structs (Go) vs Classes (JavaScript (specifically, ES6)): Ok, so I can't really comment much on ES6 Classes, as I've never worked with them. I believe these are analogues though, where we can define properties and methods belonging to our new custom data types.

Takeaway

If you come from a typeless (or dynamically typed) language, it can be a bit of a shock to have to deal with statically typed variables. I recommend spending a lot of time up front getting familiar with these concepts, since it will impact virtually every piece of code you write.

Happy hacking!
